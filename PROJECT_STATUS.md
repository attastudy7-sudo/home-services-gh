## 1. Project overview

GhanaServe is a marketplace web application connecting Ghanaian customers with local home-service professionals (plumbers, electricians, cleaners, etc.). It is built with Flask (Python 3) using SQLAlchemy, Flask-Migrate, Flask-Login, and a SQLite development database. The project is in early scaffold phase: the full database schema (17 tables) exists, all route blueprints are defined (48 routes across 5 blueprints), all templates render without errors, seed data for locations and categories is loaded, and Alembic migrations have been run against SQLite.

## 10. Business logic implemented

**Commission calculation** — `Booking.calculate_financials()` in `app/models/marketplace.py:128-133`. Reads `PLATFORM_COMMISSION_RATE` from `current_app.config`. Computes `platform_commission = agreed_price * rate` and `pro_payout_amount = agreed_price - commission`. Called in `accept_quote` (`customer/routes.py:111`).

**Quote rejection cascade** — In `accept_quote` (`customer/routes.py:114-118`), when a customer accepts one quote, all other pending quotes for the same request are bulk-updated to `status='rejected'`.

**Pro access check** — `pro_required` decorator in `pro/routes.py:16-22`. Checks `current_user.is_pro` (i.e., has a ProProfile). Redirects to `customer.dashboard` with flash if not pro. Used on 10 routes.

**Admin access check** — `admin_required` decorator in `admin/routes.py:16-22`. Checks `current_user.is_admin` (`role == 'admin'`). Aborts 403 if not admin. Used on 14 routes.

**Admin action logging** — `log_action()` helper in `admin/routes.py:25-33`. Creates an `AdminAction` record. Used in `approve_pro`, `reject_pro`, `suspend_pro`, `suspend_user`, `resolve_dispute`.

**Slug generation** — Regex `re.sub(r'[^a-z0-9]+', '-', name.lower()).strip('-')` used in two places:
- `register_pro` (`auth/routes.py:69`) — generates slug from `business_name`
- `edit_profile` (`pro/routes.py:210`) — regenerates slug when `business_name` changes
- `new_category` (`admin/routes.py:199`) — generates slug from category name

**Review average recalculation** — In `leave_review` (`customer/routes.py:181-184`). After inserting a review, fetches all reviews for the pro, recalculates `avg_rating = round(sum / count, 2)`, and updates `total_reviews`.

**Geocode caching** — `reverse_geocode` (`api/routes.py:37-45`). Uses `lat_key`/`lng_key` (rounded to 3 decimals) to check `GeocodeCache` before calling Nominatim. Cache entries are written with `UniqueConstraint('lat_key', 'lng_key')`.

**Stalled MoMo query** — `stalled_momo` (`admin/routes.py:176-181`) and used in `dashboard` (`admin/routes.py:43-48`). Filters payments where `gateway='paystack'`, `callback_received=False`, `initiated_at < now - 2 hours`, `status='pending'`.

**Type coercion helpers** — `app/helpers.py` contains 5 functions (`coerce_uuid`, `coerce_date`, `coerce_money`, `coerce_float`, `coerce_int`) used by `customer/routes.py` and `pro/routes.py` to safely convert form strings to Python types before passing to SQLAlchemy models.

**User loader** — `load_user` in `app/models/user.py:50-55`. Registered with `@login_manager.user_loader`. Converts string `user_id` to `uuid.UUID` before querying (to handle SQLite UUID mismatch).

## 12. Known issues and deferred decisions

- **SQLite type strictness** — SQLite's UUID column implementation requires Python `uuid.UUID` objects, not strings. Date columns require Python `date` objects. Workaround is the `app/helpers.py` coercion functions applied in route handlers. This will not be an issue under PostgreSQL because PostgreSQL handles type coercion for UUID and Date columns from string input.
- **`login_manager.user_loader` UUID conversion** — `load_user` in `app/models/user.py:52` calls `uuid.UUID(user_id)` on every request. This is necessary for SQLite; under PostgreSQL this conversion would not be needed.
- **`.env` uses SQLite** — The current `.env` has `DATABASE_URL=sqlite:///ghanaserve.db` for local development. The `.env.example` shows `postgresql://...`. The platform markers in `requirements.txt` exclude `psycopg2-binary` on Windows.
- **No password constraints** — `register` and `register-pro` accept any password with no minimum length, complexity, or confirmation-match validation beyond what the HTML form provides.
- **`forgot_password` is a stub** — The route always flashes a success message regardless of whether the email exists. No rate limiting, no token generation, no email sending.
- **`city_landing` uses LocationIndex slugs** — The route `/<city_slug>` uses `LocationIndex.query.filter_by(slug=city_slug).first_or_404()`. It treats the slug as an area slug, not a city name. A city-level record with `slug='accra'` was manually added to make `/accra` work.
- **No CSRF protection** — No Flask-WTF or CSRF token validation is implemented on any POST route.
- **No pagination** — All list routes return unfiltered result sets with no `.paginate()`. The admin list routes use `.limit(100)` as a cutoff.
- **No rate limiting** — No request throttling on any endpoint.
- **`post_request.html` JavaScript** — The template references JS for cascading dropdowns and geolocation but `app/static/js/main.js` is empty (`console.log` only). The feature is non-functional.
- **`GeocodeCache` requires manual population** — The `reverse-geocode` API endpoint populates the cache, but there is no bulk pre-seeding of the cache for known location centroids.
- **`reject_pro` does not remove Uniqueness constraint conflict for reused business names** — If a pro is rejected and another pro registers with the same business name, the slug generation will produce the same slug, which will violate the unique constraint on `ProProfile.slug`.
